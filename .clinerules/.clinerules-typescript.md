# Cline Rules — TypeScript (add to generic rules for TypeScript projects)

## Language & Types
- Always use TypeScript. Never use `any` — use proper types or `unknown`.
- Enable and respect strict mode. Do not use non-null assertions (`!`) unless unavoidable.
- Define interfaces in `src/types/` for shared data shapes; inline types only for local/one-off use.
- Use `Readonly<>` wrappers on props where the component does not mutate them.

## Components (React / React Native)
- Functional components only — no class components.
- Keep components focused; if a component exceeds ~200 lines, consider splitting it.
- One component per file.

## Naming
- Variables and functions: `camelCase`
- Components and types/interfaces: `PascalCase`
- Constants: `UPPER_SNAKE_CASE`
- File names: `kebab-case`
- Use British English spelling throughout.

## Imports & Paths
- Use the `@/` path alias (maps to `src/`) for all internal imports.
- Group imports: external libraries first, then internal modules, then types.

## State & Data Fetching
- Prefer `useState` and `useRef` for local state. Do not introduce a global state library without discussion.
- API utility functions should be async with explicit return types.

## Error Handling
- Wrap async operations in `try/catch`. Do not silently swallow errors.
- Surface errors to the UI or log them — never ignore them.
