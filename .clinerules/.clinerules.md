# Coding Style Rules

## Language & Types
- Always use TypeScript. Never use `any` — use proper types or `unknown`.
- Enable and respect strict mode. Do not use non-null assertions (`!`) unless unavoidable.
- Define interfaces in `src/types/` for shared data shapes; inline types only for local/one-off use.
- Use `Readonly<>` wrappers on props where the component does not mutate them.

## Components
- Functional components only — no class components.
- Mark client components with `"use client"` at the top of the file.
- Keep components focused; if a component exceeds ~200 lines, consider splitting it.
- One component per file. File names use kebab-case (e.g., `main-content.tsx`); exported components use PascalCase.

## Naming
- Variables and functions: `camelCase`
- Components and types/interfaces: `PascalCase`
- Constants: `UPPER_SNAKE_CASE`
- File names: `kebab-case`
- Do not use American English, only use British English for spelling.

## Styling
- Use Tailwind utility classes exclusively. Do not introduce CSS modules, styled-components, or inline `style` objects for layout/spacing.
- Support dark mode using `dark:` prefixes.
- Use responsive prefixes (`sm:`, `md:`, etc.) rather than JavaScript breakpoint logic.

## Imports & Paths
- Use the `@/` path alias (maps to `src/`) for all internal imports.
- Group imports: external libraries first, then internal modules, then types.

## State & Data Fetching
- Prefer `useState` and `useRef` for local state. Do not introduce a global state library without discussion.
- Use SWR for data fetching in client components.
- API utility functions live in `src/api/` and should be async with explicit return types.

## Error Handling
- Wrap async operations in `try/catch`. Do not silently swallow errors.
- Surface errors to the UI or log them — never ignore them.

## Git
- Do not create a git commit automatically. Always ask the user before committing.
- Use conventional commit format: `type(scope): message` (e.g. `feat(weather): add wind speed display`).
- Never use `--no-verify` or bypass git hooks.

## General
- Do not add comments explaining what the code does — write self-documenting code instead.
- Keep functions small and single-purpose.
- If the user rejects an edit stop what you are doing.
